### Resources changed

#### genesyscloud_auth_role

Added:
- None detected.

Removed:
- None detected.

Changed:
- `permission_policies.action_set` — When your configuration uses `"*"`, refresh and read now keep the wildcard in state instead of replacing it with the API's expanded permission list, reducing perpetual plan diffs as Genesys Cloud expands wildcard permissions.
- `permission_policies.entity_name` — When your configuration uses `"*"` for a domain, refresh and read now collapse multiple per-entity policies returned by the API back into a single wildcard policy to match your configuration.
- `default_role_id` — Updates no longer send this value to the API; changing `default_role_id` in Terraform configuration will not apply to existing roles.

#### genesyscloud_oauth_client

Added:
- None detected.

Removed:
- None detected.

Changed:
- Cascade update behavior — When an OAuth client is linked to an integration credential and the API returns an empty secret, the provider now restores the secret from the shared cache before updating the credential.
- Integration credential behavior — Creating or updating linked integration credentials now skips writing a nil client secret and logs a warning instead of failing.
- Shared OAuth cache — Cache lookups now use `clientId` and `clientSecret` keys instead of `client_id` and `client_secret`; existing cache entries written by earlier provider versions may not be found until secrets are refreshed.

### Upgrade impact
- **Auth roles with wildcard permissions:** Roles using `permission_policies` with `action_set = ["*"]` or `entity_name = "*"` should see fewer perpetual diffs after refresh; no configuration changes are required.
- **Auth role `default_role_id`:** If you rely on Terraform to change `default_role_id` on existing roles, that value is no longer sent on update; manage it outside Terraform or recreate the role.
- **OAuth clients with integration credentials:** Cascade updates handle missing secrets more reliably; verify linked credentials after upgrade if you use the shared OAuth cache file.
- **Web deployments export:** Orgs with more than one page of configurations or deployments will now see complete export results.
- Configurations not using the changed resources are likely unaffected.
